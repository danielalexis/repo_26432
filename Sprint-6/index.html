    Desenvolvimento da Sprint

    Nesta sprint foram efetuados os seguintes UC's:

    • UC1.3 - Cancelar requisição
    • UC1.4 - Modificar requisição
    • UC2.2 - Analisar requisições
    • UC2.3 - Verificar disponibilidade
    • UC3.3 - Atender solicitações de manutenção
    • UC3.4 - Registar atividades de manutenção

    UC1.3 - Cancelar Requisição

    User Stories de Caso de Uso 1.3

    • US 1.3.1: Como Aluno, Eu quero cancelar uma requisição existente, Para retirar uma solicitação previamente feita.

    Critérios para Aceitação do Caso de Uso 1.3

    • O sistema deve permitir que o aluno selecione uma requisição existente para cancelamento.
    • Após o cancelamento, o sistema deve atualizar o status da requisição para refletir o cancelamento.

    UC1.4 - Modificar Requisição

    User Stories de Caso de Uso 1.4

    • US 1.4.1: Como Aluno, Eu quero modificar os detalhes de uma requisição existente, Para ajustar as informações conforme necessário.

    Critérios para Aceitação do Caso de Uso 1.4

    • O sistema deve permitir que o aluno selecione uma requisição existente para modificação.
    • Após a modificação, o sistema deve atualizar os detalhes da requisição conforme fornecido pelo aluno.
    • Caso a requisição esteja em processo de análise ou execução, o sistema não deve deixar editar a requisição.

    UC2.2 - Analisar Requisições

    User Stories de Caso de Uso 2.2

    • US 2.2.1: Como Proprietário, Eu quero analisar as requisições dos alunos, Para tomar decisões informadas sobre sua execução.

    Critérios para Aceitação do Caso de Uso 2.2

    • O sistema deve apresentar uma lista de requisições pendentes para análise pelo proprietário.
    • Durante a análise, o proprietário deve poder visualizar detalhes relevantes, como descrição da requisição e data solicitada.
    • O proprietário deve ter a opção de aprovar ou rejeitar uma requisição, com feedback opcional para o aluno.

    UC2.3 - Verificar Disponibilidade

    User Stories de Caso de Uso 2.3

    • US 2.3.1: Como Proprietário, Eu quero verificar a disponibilidade de recursos solicitados pelos alunos, Para garantir que as requisições sejam atendidas adequadamente.

    Critérios para Aceitação do Caso de Uso 2.3

    • O sistema deve fornecer uma visualização da disponibilidade de recursos, considerando a demanda atual.
    • Durante a análise de uma requisição, o proprietário deve ser informado sobre a disponibilidade de recursos na data solicitada.
    • Se os recursos não estiverem disponíveis, o sistema deve oferecer alternativas ou permitir que o proprietário proponha uma data alternativa.

    UC3.3 - Atender Solicitações de Manutenção

    User Stories de Caso de Uso 3.3

    • US 3.3.1: Como Técnico de Manutenção, Eu quero atender às solicitações de manutenção, Para realizar as atividades necessárias.

    Critérios para Aceitação do Caso de Uso 3.3

    • O sistema deve apresentar uma lista de solicitações de manutenção pendentes para o técnico.
    • Durante o atendimento, o técnico deve poder visualizar detalhes relevantes, como tipo de manutenção e localização.
    • O técnico deve ter a opção de marcar a solicitação como atendida, registando a conclusão da atividade.

    UC3.4 - Registrar Atividades de Manutenção

    User Stories de Caso de Uso 3.4

    • US 3.4.1: Como Técnico de Manutenção, Eu quero registrar as atividades realizadas durante a manutenção, Para manter um histórico detalhado.

    Critérios para Aceitação do Caso de Uso 3.4

    • Após atender a uma solicitação de manutenção, o técnico deve poder registrar detalhes sobre as atividades realizadas.
    • O sistema deve armazenar as informações registradas, incluindo data, descrição das atividades e observações relevantes.
    • O histórico de atividades deve estar disponível para consulta futura, fornecendo um registro completo das manutenções realizadas.
